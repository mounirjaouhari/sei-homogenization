## PROG-GC-UNITS — fracture energy off by ~2–3 orders of magnitude (units error)

**Provable.** Theorem (Gc) is `Gc = eps_SEI * sum_i f_i g_i`. For `Gc` to be in
J/m², `g_i` must be a **volumetric** density [J/m³] (since [m]·[J/m³]=[J/m²]).
The code hard-coded `g_i = 5e9/3e9/0.5e9 J/m³`, giving

```
g_avg = 0.3·5e9 + 0.4·3e9 + 0.3·0.5e9 = 2.85e9 J/m³
Gc(30 nm) = 30e-9 · 2.85e9 = 85.5 J/m²      (≈ 280× the article's Table 3 = 0.18)
```

The committed CSV likewise has `Gc = 22 / 98.25 / 195.25 J/m²`. Internally,
`config_parameters.m` even defined a *reasonable* `gc_volumetric = 5e6 J/m³` that
was then **ignored**.

**Fix (applied).** `g_i` is now derived, reproducibly, from each constituent's
strength and modulus, `g_i = σ_{r,i}²/(2 E_i)` [J/m³] (the elastic energy density
at the strength limit), removing the magic `1e9`:

```
g_LiF = (250e6)²/(2·65e9) = 4.81e5 ;  g_Li2CO3 = 3.33e5 ;  g_org = 2.25e5  [J/m³]
Gc(30 nm) ≈ 0.0114 J/m²
```

**Modeling choice (not imposed).** *Which* definition of `g_i` to adopt is a
convention. The σ_r²/(2E) choice is dimensionally consistent and reproducible
from `constituents_properties.csv`, and it matches the skills. Matching the
article's Table 3 magnitude (~0.05–0.32 J/m²) instead would require a different,
also-legitimate `g_i` (~5e6 J/m³). The code is now **internally consistent**
(theory = MATLAB = Python core = corrected CSV); the authors decide the final
magnitude on the article side.

## PROG-DEFF-BASE — inconsistent power-law base ⇒ nonsensical `D_eff`

**Provable.** `K_eff` and `R_eff` used the dimensionless base `(eps/L)`, but
`D_eff = eps^γ · D_sei` used the **dimensional** thickness. A power law must use a
single dimensionless base; mixing them gave

```
D_eff(buggy) ≈ 4.7e-28 … 9.7e-23 m²/s     (physically impossible diffusivity)
```

**Fix (applied).** All four effective parameters now use `eta = eps/L`:
`D_eff = eta^γ · D_sei` (and `beta_eff = eta^δ·…`), giving base-consistent
`D_eff ≈ 1.3e-21 … 3.4e-19 m²/s`.

## PROG-PY-MISSING-SRC — the Python pipeline cannot run

**Provable.** `main.py` and `example_usage.py` do
`from config import …`, `from homogenize import …`, `from compute_effective import …`,
`from generate_matlab/latex/plot_composition import …`, all from a `src/`
package that **is not present** in `programmes/`. Importing fails immediately.

**Resolution.** Rather than guess the intended API of seven missing modules
(including a Materials-Project network fetch and LaTeX/figure generators), a
self-contained, **runnable** numerical core is provided:

```
python recompute_effective.py        # regenerates effective_parameters_corrected.csv
```

It reuses the verified `Skills/sei_physics.py` (single source of truth) and prints
a before/after comparison. Restoring the full `src/` package (MP fetch, table/
figure generation) remains a separate task if those outputs are needed.

## PROG-COMMENT — misleading comment (fixed)

`config_parameters.m` claimed "le facteur eps^alpha est absorbé dans la définition
de E_sei" while the code explicitly applied `(eps/L)^(alpha-1)`. The comment now
matches the code.

## PROG-ARCH-MISMATCH — documented architecture ≠ actual layout

`main_validation.m` `addpath`s subfolders (`config/ mesh/ assembly/ solve/ …`) that
do not exist; all `.m` files are flat in `programmes/`. Harmless if run from
`programmes/`, but the documented architecture should be reconciled with the flat
layout (or the files moved into the subfolders).

## Methodological guarantee

No correction forces a scientifically unprovable result. Constituent values are
literature mid-ranges; `g_i` is derived transparently from them. The only facts
asserted are dimensional inconsistencies, an inconsistent power-law base, a
non-importable package, and a stale CSV — all provable.
